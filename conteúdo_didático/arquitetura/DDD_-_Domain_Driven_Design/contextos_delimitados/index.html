    O que é Contexto Delimitado

    O Contexto Delimitado é um conceito central no Domain-Driven Design (DDD). Ele se refere a uma delimitação clara de um modelo de domínio específico, onde as palavras, conceitos e regras têm significados precisos e consistentes. Cada contexto delimitado define um limite em torno de um modelo de domínio e estabelece a linguagem e as interações que ocorrem dentro desse limite.

    Principais Características do Contexto Delimitado

    1. Modelo de Domínio Específico:

    2. Dentro de um contexto delimitado, você pode ter um modelo de domínio que é específico para aquela parte do sistema. Isso significa que as entidades e serviços têm definições e comportamentos que são relevantes apenas dentro desse contexto.

    3. Linguagem Ubiqua:

    4. O contexto delimitado incentiva o uso de uma linguagem ubíqua (ou Ubiquitous Language), que é uma linguagem comum usada por todos os membros da equipe para descrever o modelo de domínio, ajudando a evitar ambiguidades e mal-entendidos.

    5. Limite Claro:

    6. Cada contexto delimitado possui limites claros que definem onde um modelo de domínio se aplica. Fora desses limites, o mesmo conceito pode ter um significado diferente ou pode não ser aplicável.

    Entidades em Múltiplos Contextos

    Uma mesma entidade pode aparecer em diferentes contextos delimitados, mas com significados e contratos distintos. Vamos usar o exemplo do Produto para ilustrar isso:

    1. Contexto Delimitado: Catálogo de Produtos:

    2. No contexto do Catálogo, o Produto pode incluir atributos relacionados ao marketing, como descrição, imagens, e preços promocionais. Este contexto é voltado para como o produto é apresentado aos clientes.

    3. Contexto Delimitado: Pedidos:

    4. No contexto de Pedidos, o Produto é tratado como um Item de Pedido. Aqui, ele pode incluir informações como quantidade e preço no momento da compra, que podem ser diferentes das informações do catálogo.

    5. Contexto Delimitado: Pagamentos:

    6. No contexto de Pagamentos, o Produto pode ser representado como parte do cálculo total do valor da transação. Este contexto foca nos aspectos financeiros e pode ter informações sobre descontos ou taxas.

    Mapeamento de Entidades

    As mesmas entidades podem ser mapeadas de várias maneiras entre os contextos:

    • Tabela Única:

    • Em alguns casos, as entidades podem ser armazenadas na mesma tabela em um banco de dados. Por exemplo, a tabela de produtos poderia conter todas as informações relevantes para o catálogo, pedidos e pagamentos, mas com colunas que refletem as diferentes necessidades de cada contexto.

    • Tabelas Independentes:

    • Alternativamente, cada contexto pode ter sua própria tabela para armazenar dados específicos. Isso permite uma maior flexibilidade e especialização, garantindo que cada contexto atenda às suas necessidades sem ser afetado por mudanças em outros contextos.
    • Nesse caso, pode haver replicação de dados entre os contextos, onde um serviço ou mecanismo de sincronização é responsável por garantir que as informações relevantes sejam atualizadas em todos os contextos.

    Interação entre Contextos Delimitados

    Quando diferentes contextos delimitados precisam interagir, é importante definir as interfaces e contratos que permitirão essa comunicação. Por exemplo, ao criar um pedido, o contexto de Pedidos pode precisar consultar o contexto de Catálogo para obter informações sobre o produto, mas isso deve ser feito através de uma API bem definida.

    Vantagens do Uso de Contextos Delimitados

    1. Clareza e Consistência:

    2. A definição de contextos delimitados ajuda a manter a clareza e a consistência no modelo de domínio, evitando ambiguidade nas definições.

    3. Facilidade de Manutenção:

    4. Com limites claros definidos, as equipes podem trabalhar em diferentes contextos sem interferir nas outras partes do sistema, facilitando a manutenção e a evolução do software.

    5. Escalabilidade:

    6. Contextos delimitados permitem que sistemas complexos sejam divididos em partes menores e mais gerenciáveis, facilitando a escalabilidade e a implementação de novas funcionalidades.

    O que é Camada Anti-Corrupção (Anti-Corruption Layer - ACL)

    A Camada Anti-Corrupção é um padrão arquitetural utilizado no Domain-Driven Design (DDD) para evitar que um contexto delimitado seja "corrompido" por influências externas ou por outros contextos. Essa camada atua como um intermediário que traduz as interações entre contextos delimitados, permitindo que eles se comuniquem sem que um contexto influencie negativamente o modelo de domínio do outro.

    Principais Características da Camada Anti-Corrupção

    1. Isolamento:

    2. A camada anti-corrupção isola o modelo de domínio de um contexto de qualquer mudança ou comportamento inesperado que possa vir de outro contexto. Isso é especialmente importante em sistemas complexos onde diferentes contextos têm suas próprias regras e modelos.

    3. Tradução de Modelos:

    4. Quando dois contextos delimitados interagem, pode haver diferenças significativas nas representações de dados e na semântica. A camada anti-corrupção atua como um tradutor, convertendo dados e chamadas de método de um contexto para um formato que o outro contexto pode entender.

    5. Encapsulamento de Lógica de Integração:

    6. A lógica necessária para integrar e comunicar entre contextos é encapsulada dentro da camada anti-corrupção, tornando mais fácil modificar ou substituir a lógica de integração sem impactar os modelos de domínio dos contextos.

    Exemplo de Camada Anti-Corrupção

    Vamos usar um exemplo de um sistema de e-commerce com dois contextos delimitados: Catálogo de Produtos e Pedidos.

    Cenário

    • Contexto Catálogo de Produtos:

    • Este contexto possui uma representação rica dos produtos, com informações detalhadas sobre marketing, disponibilidade, etc.

    • Contexto Pedidos:

    • Este contexto precisa criar pedidos com informações de produtos, mas não deve depender da estrutura ou lógica do contexto de Catálogo de Produtos.

    Implementação da Camada Anti-Corrupção

    A camada anti-corrupção pode ser implementada como um serviço ou um módulo que traduz as informações do contexto de Catálogo para o contexto de Pedidos. Por exemplo:

    class ProdutoCatalogo:
    def __init__(self, id, nome, preco, descricao):
        self.id = id
        self.nome = nome
        self.preco = preco
        self.descricao = descricao

class PedidoService:
    def __init__(self, catalogo_service):
        self.catalogo_service = catalogo_service

    def criar_pedido(self, produto_id):
        # Usar a camada anti-corrupção para obter informações do catálogo
        produto_info = self.catalogo_service.obter_produto(produto_id)

        # Converter para o formato do pedido
        item_pedido = {
            "produto_id": produto_info.id,
            "nome": produto_info.nome,
            "preco": produto_info.preco,
            # Outros atributos necessários para o pedido
        }
        # Lógica para adicionar o item ao pedido
        return item_pedido

class CatalogoService:
    def obter_produto(self, produto_id):
        # Simula a obtenção de um produto do Catálogo
        return ProdutoCatalogo(produto_id, "Produto A", 100.0, "Descrição do Produto A")

    Vantagens da Camada Anti-Corrupção

    1. Proteção do Modelo de Domínio:

    2. A camada anti-corrupção protege o modelo de domínio do contexto de ser afetado por mudanças ou inconsistências em outros contextos, garantindo que ele permaneça puro e consistente.

    3. Facilidade de Mudanças:

    4. Se um contexto externo muda (por exemplo, o formato de dados ou a lógica de negócios), as alterações podem ser feitas apenas na camada anti-corrupção, minimizando o impacto em todo o sistema.

    5. Comunicação Clara:

    6. A camada define uma interface clara para a comunicação entre contextos, facilitando a compreensão das interações e promovendo um design mais limpo.

    7. Facilidade de Testes:

    8. A lógica de integração e tradução é encapsulada, permitindo que você teste a camada anti-corrupção independentemente dos contextos que ela conecta.
