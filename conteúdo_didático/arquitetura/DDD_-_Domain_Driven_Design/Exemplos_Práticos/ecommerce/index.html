    E-Commerce

    Modelo de Domínio

    Para construir o domínio de um sistema de e-commerce que abrange o fluxo de compras do cliente final, precisamos identificar as principais entidades de negócio que compõem esse fluxo. Aqui está uma descrição de cada uma dessas entidades, seu papel no sistema e como elas se relacionam entre si.

    1. Cliente

    O cliente é a entidade que representa o usuário final que está fazendo compras no e-commerce. Essa entidade deve armazenar informações como nome, e-mail, endereço e dados de contato.

    2. Produto

    O produto é a unidade básica de venda no e-commerce. Cada produto deve ter informações como nome, descrição, preço, categoria e quantidade disponível. O produto pode estar listado em um catálogo.

    3. Catálogo

    O catálogo é uma coleção de produtos disponíveis para venda. Ele permite que os clientes naveguem e filtrem produtos com base em categorias, preços e outros critérios. O catálogo pode ser considerado uma visualização dos produtos disponíveis.

    4. Carrinho

    O carrinho representa uma coleção temporária de produtos que o cliente deseja comprar. O cliente pode adicionar, remover ou alterar a quantidade de produtos no carrinho antes de finalizar a compra. O carrinho deve armazenar referências aos produtos e a quantidade escolhida de cada um.

    5. Pedido

    Um pedido é gerado quando o cliente decide finalizar a compra. Ele representa a transação e deve incluir informações sobre o cliente, os produtos comprados, o total a ser pago e o status do pedido. O pedido é a entidade que persiste as informações da compra.

    6. Pagamento

    A entidade de pagamento gerencia as informações relacionadas ao processo de pagamento do pedido. Ela deve armazenar informações como método de pagamento, status do pagamento, data e valor.

    Raciocínio para Montar o Domínio

    • Identificação das Entidades: Iniciamos identificando as entidades fundamentais que participam do fluxo de compra. Cada entidade deve refletir uma parte do processo de compra que é relevante para o cliente.

    • Relacionamentos: Após identificar as entidades, mapeamos os relacionamentos entre elas. Por exemplo, um cliente pode ter múltiplos pedidos, um pedido pode ter múltiplos produtos, e cada produto pode estar em muitos pedidos (relação muitos-para-muitos).

    • Atributos: Para cada entidade, definimos os atributos necessários que permitirão que o sistema funcione corretamente e que capturem todas as informações relevantes.

    • Coesão e Responsabilidade: Nos asseguramos de que cada entidade tenha uma única responsabilidade e que seus atributos e comportamentos sejam coesos, proporcionando uma melhor manutenção e entendimento do sistema.

    Modelo de infraestrutura Repositórios e Serviços

    1. Separação de Responsabilidades:

    2. Repositórios: A camada de repositórios é responsável por todas as interações com a base de dados. Isso permite que a lógica de acesso a dados seja isolada, facilitando futuras mudanças na forma como os dados são armazenados ou recuperados (por exemplo, a troca de uma base de dados relacional por uma NoSQL).

    3. Serviços: A camada de serviços contém a lógica de negócio que orquestra as operações necessárias para atender às requisições do cliente. Ao centralizar a lógica de negócio, garantimos que as regras do domínio sejam aplicadas de maneira consistente em toda a aplicação.

    4. Fluxo de Compras:

    5. Ao adicionar um produto ao carrinho, o controlador chamará o serviço correspondente, que, por sua vez, utilizará o repositório para persistir as informações do produto no carrinho.

    6. Quando um cliente finaliza um pedido, o serviço de pedidos irá coordenar a criação do pedido, interagir com a camada de repositórios para salvar o pedido na base de dados e também calcular o total do pedido, utilizando as informações dos itens e do cliente.

    7. O serviço de pagamento orquestrará a lógica de processamento do pagamento, chamando o repositório quando necessário para registrar o status do pagamento.

    8. Manutenção e Testabilidade:

    9. A utilização dessas camadas permite que a lógica de negócios (serviços) seja testada isoladamente, sem depender da implementação da persistência (repositórios). Isso é fundamental para garantir que as regras de negócio sejam verificáveis em testes unitários.

    10. Mudanças na persistência dos dados (como a troca de um banco de dados) exigirão apenas alterações na camada de repositórios, enquanto a lógica de negócios permanecerá intacta.

    Uso dos Controllers

    1. Responsabilidade dos Controllers:

    2. Os Controllers são responsáveis por receber as requisições HTTP e retornar as respostas apropriadas. Eles atuam como intermediários entre o cliente e a lógica de negócios da aplicação.

    3. Cada controller deve estar associado a uma entidade ou um conjunto de funcionalidades específicas, como gerenciamento de produtos, clientes, pedidos e pagamentos.

    4. Interação com os Services:

    5. Quando uma requisição é recebida, o controller invoca o serviço apropriado para processar a lógica de negócios necessária. Por exemplo, ao criar um novo produto, o controlador de produtos chamará o ProductService.createProduct().

    6. O controller deve lidar com a validação das entradas e formatar as respostas, enquanto a lógica de negócios complexa é delegada ao serviço.

    7. Manutenção e Testabilidade:

    8. A separação entre controllers e serviços permite que ambos sejam testados de forma independente. Os controllers podem ser testados utilizando mocks para os serviços, garantindo que a interface HTTP funcione conforme o esperado.

    9. Qualquer alteração na lógica de negócios não afetará diretamente os controllers, desde que a interface do serviço permaneça a mesma.

    10. Fluxo de Operações:

    11. Quando um cliente realiza uma ação (como adicionar um produto ao carrinho), a requisição é dirigida ao controller correspondente. O controller chama o serviço apropriado, que realiza as operações necessárias e interage com os repositórios conforme necessário.

    12. O resultado é retornado ao controller, que então formata a resposta e a envia de volta ao cliente.

    13. Operações:

    14. Buscar Produtos No Catalogo: Retorna uma lista de produtos disponíveis no catálogo.

    15. Adicionar Produto Ao Carrinho: Adiciona um produto ao carrinho do cliente.

    16. Calcular Desconto Carrinho: Aplica um desconto ao carrinho do cliente e retorna o carrinho atualizado.

    17. Realizar Pedido Cria um novo pedido com os produtos no carrinho do cliente e retorna o pedido gerado.

    18. Pagar Pedido: Processa o pagamento para o pedido especificado e retorna os detalhes do pagamento.
