    Python Backend Com FastAPI e Peewee

    Teoria

    O projeto Python Backend é uma aplicação web construída com FastAPI, um framework moderno e de alto desempenho para construir APIs com Python 3. Ele utiliza Peewee como ORM (Object-Relational Mapping) para interagir com um banco de dados PostgreSQL, e dotenv para gerenciar variáveis de ambiente.

    Utilidade

    • FastAPI: Framework para construção de APIs RESTful de forma rápida e eficiente, com suporte a validação de dados e documentação automática.

    • Peewee: ORM leve e simples que facilita a interação com bancos de dados, permitindo manipulação de dados através de modelos.

    • psycopg2-binary: Driver para conectar aplicações Python ao PostgreSQL, permitindo executar consultas e manipular dados no banco.

    • python-dotenv: Carrega variáveis de ambiente a partir de um arquivo .env, facilitando a configuração de dados sensíveis como credenciais.

    Exemplo de Uso

    1.1. Estrutura do Projeto:

    /python-backend
│
├── app
│   ├── __init__.py
│   ├── main.py
│   ├── models.py
│   ├── repositories.py
│   ├── services.py
│   └── .env
│
├── tests
│   └── test_app.py
│
├── Dockerfile
├── docker-compose.yml
└── pyproject.toml

    2. FastAPI

    Teoria

    FastAPI é um framework para a construção de APIs com Python, projetado para ser rápido e fácil de usar. Ele utiliza a tipagem do Python para fornecer validação automática de dados e documentação interativa.

    Utilidade

    • Desempenho Alto: Construído sobre Starlette e Pydantic, o FastAPI é capaz de lidar com várias requisições simultaneamente, tornando-o ideal para aplicações de alto desempenho.

    • Documentação Automática: Gera automaticamente uma documentação interativa da API em Swagger e ReDoc.

    • Validação de Dados: Utiliza tipos de dados do Python para validar automaticamente as entradas.

    Exemplo de Uso

    from fastapi import FastAPI

app = FastAPI()

@app.get("/")
def read_root():
    return {"Hello": "World"}

    3. Peewee

    Teoria

    Peewee é um ORM (Object-Relational Mapping) leve e simples para Python. Ele fornece uma interface intuitiva para interagir com bancos de dados, permitindo que desenvolvedores manipulem dados como objetos Python.

    Utilidade

    • Facilidade de Uso: A sintaxe é simples e intuitiva, permitindo operações de banco de dados com menos código.

    • Suporte a Múltiplos Bancos de Dados: Funciona com SQLite, PostgreSQL e MySQL, entre outros.

    • Modelos de Dados: Permite a definição de modelos de dados de forma declarativa, facilitando a manipulação de registros.

    Exemplo de Uso

    from peewee import Model, CharField, SqliteDatabase

db = SqliteDatabase('my_database.db')

class User(Model):
    username = CharField()
    email = CharField()

    class Meta:
        database = db

db.connect()
db.create_tables([User])

    4. psycopg2-binary

    Teoria

    psycopg2-binary é um adaptador PostgreSQL para Python. Ele permite que aplicações Python se conectem a um banco de dados PostgreSQL e executem operações SQL.

    Utilidade

    • Conexão com PostgreSQL: Facilita a conexão e execução de comandos SQL em bancos de dados PostgreSQL.

    • Suporte a Transações: Permite gerenciar transações de forma eficiente, garantindo a integridade dos dados.

    • Desempenho: É otimizado para desempenho e pode lidar com operações em massa de forma eficiente.

    Exemplo de Uso

    import psycopg2

conn = psycopg2.connect("dbname=mydatabase user=myuser password=mypassword")
cur = conn.cursor()
cur.execute("SELECT * FROM my_table;")
rows = cur.fetchall()
cur.close()
conn.close()

    5. python-dotenv

    Teoria

    python-dotenv é uma biblioteca que carrega variáveis de ambiente a partir de um arquivo .env. É útil para gerenciar configurações sensíveis, como credenciais de banco de dados, sem expô-las no código.

    Utilidade

    • Gerenciamento de Configurações: Permite separar configurações sensíveis do código-fonte, facilitando a manutenção.

    • Facilidade de Uso: Carrega as variáveis de ambiente automaticamente, tornando-as acessíveis via os.environ.

    • Compatibilidade: Funciona bem com qualquer aplicação Python que precise de variáveis de ambiente.

    Exemplo de Uso

    from dotenv import load_dotenv
import os

load_dotenv()
database_url = os.getenv("DATABASE_URL")

    6. Testes com pytest

    Teoria

    pytest é um framework de testes para Python que facilita a escrita de testes simples e escaláveis. Ele suporta testes de unidade, funcionais e integração.

    Utilidade

    • Simplicidade: Sintaxe simples que permite escrever testes de forma rápida e intuitiva.

    • Plugins: Possui uma vasta gama de plugins que estendem suas funcionalidades.

    • Testes Assíncronos: Suporta testes assíncronos, permitindo testar código que utiliza asyncio.

    Exemplo de Uso

    import pytest

def test_example():
    assert 1 + 1 == 2

    Como aplicar os conceitos de arquitetura com essa stack?

    1. Domain-Driven Design (DDD)

    Teoria

    O DDD é uma abordagem de design de software que foca na colaboração entre especialistas de domínio e desenvolvedores, visando criar um modelo que represente com precisão as complexidades do negócio.

    Como a Stack Ajuda

    • Modelagem de Domínio com Peewee: O Peewee permite definir modelos de dados que refletem diretamente os conceitos do domínio. Cada tabela do banco de dados pode ser representada como uma classe de modelo, facilitando a manipulação e a lógica de negócios.

    Exemplo de Modelagem:

    from peewee import Model, CharField, TextField, ForeignKeyField, SqliteDatabase

db = SqliteDatabase('feedback.db')

class Feedback(Model):
    content = TextField()

    class Meta:
        database = db

class Question(Model):
    text = CharField()
    feedback = ForeignKeyField(Feedback, backref='questions')

    class Meta:
        database = db
    • Separação de Preocupações: A estrutura do projeto (com pastas como repositories, services, etc.) ajuda a organizar a lógica de negócios separadamente dos detalhes de implementação. Isso é fundamental no DDD.

    Exemplo de Repositório:

    from .models import Feedback

def create_feedback(content: str) -> Feedback:
    return Feedback.create(content=content)

def get_feedback(feedback_id: int) -> Feedback:
    return Feedback.get_by_id(feedback_id)

    2. Test-Driven Development (TDD)

    Teoria

    O TDD é uma prática de desenvolvimento onde os testes são escritos antes do código de produção, garantindo que o código atenda aos requisitos definidos desde o início.

    Como a Stack Ajuda

    • Uso do pytest: O pytest é uma ferramenta poderosa para escrever testes. Sua simplicidade e flexibilidade permitem que desenvolvedores escrevam testes de unidade e integração rapidamente, incentivando a prática de TDD.

    • Uso do TestClient: O TestClient do FastAPI permite simular requisições HTTP durante os testes, facilitando a verificação do comportamento da API. Isso ajuda a garantir que a lógica de negócios funcione conforme esperado.

    Exemplo de Teste com TestClient:

    import pytest
from fastapi.testclient import TestClient
from app.main import app

client = TestClient(app)

def test_create_feedback():
    response = client.post("/feedback/", json={"content": "Test feedback"})
    assert response.status_code == 200
    assert response.json()["content"] == "Test feedback"
    • Ciclo Red-Green-Refactor: Com pytest e TestClient, você pode facilmente seguir o ciclo TDD (escrever um teste que falha, implementar o código para passar o teste e refatorar). Isso ajuda a garantir que o código esteja sempre coberto por testes.

    Exemplo de Ciclo TDD:

    1. Escrever um teste que falha:
    def test_get_feedback_not_found():
    response = client.get("/feedback/999")  # ID que não existe
    assert response.status_code == 404
    1. Implementar o código para passar o teste:
    @app.get("/feedback/{feedback_id}")
def get_feedback(feedback_id: int):
    try:
        return service_get_feedback(feedback_id)
    except Feedback.DoesNotExist:
        raise HTTPException(status_code=404, detail="Feedback not found")
    1. Refatorar o código se necessário.

    3. SOLID

    Teoria

    Os princípios SOLID são diretrizes para o design de software que visam tornar o código mais compreensível, flexível e manutenível.

    Como a Stack Ajuda

    • Single Responsibility Principle (SRP): A estrutura do projeto, com diferentes pastas para models, repositories e services, promove a separação de responsabilidades.

    Exemplo de SRP:

    # models.py
class Feedback(Model):
    content = TextField()

# repositories.py
def create_feedback(content: str) -> Feedback:
    return Feedback.create(content=content)

# services.py
def service_create_feedback(content: str) -> Feedback:
    return create_feedback(content)
    • Open/Closed Principle (OCP): As classes de modelo e os serviços podem ser facilmente estendidos sem modificar o código existente.

    Exemplo de OCP:

    # Adicionando um novo método no repositório sem modificar o existente
def get_all_feedback() -> List[Feedback]:
    return list(Feedback.select())
    • Liskov Substitution Principle (LSP): Ao usar interfaces ou classes base quando necessário, você pode garantir que subclasses possam ser usadas como substitutas de suas classes base.

    Exemplo de LSP:

    class BaseRepository:
    def create(self, data):
        pass

class FeedbackRepository(BaseRepository):
    def create(self, content: str) -> Feedback:
        return Feedback.create(content=content)
    • Interface Segregation Principle (ISP): A estrutura modular do projeto permite que você crie interfaces pequenas e específicas.

    Exemplo de ISP:

    class FeedbackServiceInterface:
    def create_feedback(self, content: str) -> Feedback:
        pass

class FeedbackService(FeedbackServiceInterface):
    def create_feedback(self, content: str) -> Feedback:
        return create_feedback(content)
    • Dependency Inversion Principle (DIP): Manter a lógica de negócios separada da implementação concreta, permitindo fácil troca de implementações.

    Exemplo de DIP:

    class FeedbackService:
    def __init__(self, repository: FeedbackRepository):
        self.repository = repository

    def create_feedback(self, content: str) -> Feedback:
        return self.repository.create(content)
